## 1. PRD (Product Requirements Document)

### 한 줄 정의
> **"우리가 무엇을 왜 만드는가"를 팀 전체가 공유하기 위한 문서**

개발자, 디자이너, 기획자, 경영진이 **같은 이해, 같은 방향**으로 일하게 만드는 설계도입니다.

### 왜 필요한가?

PRD가 없으면 이런 일이 생깁니다.

```
기획자  → "버튼 누르면 팝업 띄워주세요"
개발자  → (어떤 버튼? 조건은? 데이터는?)
디자이너 → (크기는? 모바일은? 닫기 버튼은?)
QA      → (실패하면? 예외 케이스는?)
```

PRD는 이 모든 질문에 **미리 답**해두는 문서입니다.

### PRD 핵심 구성 8가지

| # | 섹션 | 핵심 질문 | 예시 |
|---|------|----------|------|
| 1 | **배경/목적** | 왜 만드는가? | 회원가입 이탈률 43% → 소셜 로그인으로 해결 |
| 2 | **대상 사용자** | 누가 쓰는가? | 30대 직장인, 모바일 신규 유저 |
| 3 | **성공 지표** | 성공을 어떻게 측정? | 가입 완료율 57%→70%, 소요시간 3분→1분 |
| 4 | **기능 요구사항** | 무엇을 만드는가? | Must/Should/Nice로 우선순위 구분 |
| 5 | **범위 제외** | 무엇을 안 하는가? | 페이스북 로그인은 다음 분기 |
| 6 | **기능 명세** | 구체적으로 어떻게? | 플로우, 예외처리 케이스 상세 기술 |
| 7 | **UX/UI 요구사항** | 화면은 어떻게? | Figma 링크, 반응형 여부, 접근성 |
| 8 | **일정/의존성** | 언제까지? 뭐가 필요? | 디자인 1주, 개발 2~3주, QA 4주차 |

### 좋은 PRD vs 나쁜 PRD

| 나쁜 PRD ❌ | 좋은 PRD ✅ |
|-----------|-----------|
| "빠르게 로딩되어야 한다" | "첫 화면 LCP 2초 이내" |
| "사용하기 쉬워야 한다" | "튜토리얼 없이 3분 내 첫 작업 완료" |
| "에러 처리 잘 하기" | "네트워크 오류 시 재시도 버튼 + 메시지 표시" |
| 해결책부터 기술 | 문제 정의 → 해결책 순서로 작성 |

## 2. MCP 서버 — 설치 후 연결을 통해 기능 사용

### 개념

MCP(Model Context Protocol)는 Claude가 외부 도구·서비스와 연결되는 **표준 규격**입니다.  
**설치 → 연결 → 기능 사용** 순서로 동작합니다.

```
설치 (Install)         연결 (Connect)         사용 (Use)
─────────────          ──────────────          ──────────
npx로 서버 실행    →   config에 등록      →   Claude가 도구 호출
또는 로컬 실행         Claude가 인식            자동으로 실행
```

### 동작 3단계 상세

**① 설치** — MCP 서버를 실행 가능한 상태로 만들기
```json
{
  "mcpServers": {
    "shadcn": {
      "command": "npx",
      "args": ["shadcn@latest", "mcp"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "ghp_xxxx" }
    }
  }
}
```

**② 연결** — Claude가 시작 시 MCP 서버를 자동 인식
```
Claude 시작
  → config 읽기
  → 각 MCP 서버 프로세스 실행
  → 사용 가능한 도구 목록 수신
  → "이제 shadcn, github 도구를 쓸 수 있음" 인식 완료
```

**③ 기능 사용** — 자연어로 요청 → 자동 도구 호출
```
개발자: "Button 컴포넌트 추가해줘"
  → Claude가 shadcn MCP의 add_component 도구 호출
  → 실제로 파일이 프로젝트에 생성됨
```

### MCP가 할 수 있는 것

| 종류 | 예시 |
|------|------|
| **읽기** | 파일 읽기, 문서 조회, DB 검색 |
| **실행** | CLI 명령, API 호출, 컴포넌트 설치 |
| **쓰기** | 파일 생성/수정, DB INSERT, PR 생성 |

## 3. MVP (Minimum Viable Product)

### 한 줄 정의
> **"가장 적은 기능으로 가장 빠르게 시장 반응을 검증하는 첫 번째 제품"**

### 핵심 철학

```
❌ 잘못된 이해
완벽한 제품의 축소판 → 기능을 다 만들되 퀄리티를 낮춤

✅ 올바른 이해
핵심 가설을 검증할 수 있는 최소 단위
→ "사람들이 이 문제를 해결하려고 돈을 낼까?"에 답하는 것
```

### MVP 결정 기준

```
이 기능이 없으면 → 핵심 가치를 전달할 수 없는가?
  YES → MVP에 포함
  NO  → MVP에서 제외 (나중에)
```

### 실제 예시 — 배달 앱 MVP

| 단계 | 포함 여부 | 이유 |
|------|----------|------|
| 음식 주문 | ✅ Must | 핵심 기능 |
| 결제 | ✅ Must | 없으면 서비스 불가 |
| 실시간 배달 추적 | ❌ 제외 | 전화로 대체 가능 |
| 리뷰/평점 | ❌ 제외 | 검증 후 추가 |
| 쿠폰/이벤트 | ❌ 제외 | 나중에 |

### PRD와 MVP의 관계

```
PRD → 전체 제품의 설계도 (큰 그림)
MVP → PRD 중에서 지금 당장 만들 핵심만 추린 것
```

## 4. Sub-agent / Agent Team / Task Manager — 개념과 차이

### 개념 정의

**Sub-agent (서브 에이전트)**
> 메인 AI가 특정 작업을 위임하는 **전문화된 하위 AI**

```
메인 Claude
  ├─ Sub-agent A: "코드 작성 전담"
  ├─ Sub-agent B: "테스트 전담"
  └─ Sub-agent C: "문서화 전담"
```

**Agent Team (에이전트 팀)**
> 여러 Sub-agent가 **협력해서 하나의 큰 목표**를 달성하는 구조

```
목표: "새 기능 개발"

Agent Team
  ┌─ Researcher Agent  → 요구사항 분석
  ├─ Coder Agent       → 코드 구현
  ├─ Reviewer Agent    → 코드 리뷰
  └─ Tester Agent      → 테스트 실행
         ↓
    최종 결과물 통합
```

**Task Manager (태스크 매니저)**
> Agent Team이 해야 할 **일의 목록과 순서, 상태를 관리**하는 시스템

```
Task Manager가 하는 일:
  - 큰 작업을 작은 태스크로 분해
  - 각 태스크를 어느 Agent에게 줄지 결정
  - 완료/실패/대기 상태 추적
  - 의존성 관리 (A 끝나야 B 시작)
```

### 세 개념의 관계도

```
┌─────────────────────────────────────┐
│           Task Manager              │  ← 전체 오케스트라 지휘자
│  [Task1] [Task2] [Task3] [Task4]   │
└──────────────┬──────────────────────┘
               │ 태스크 배분
┌──────────────▼──────────────────────┐
│           Agent Team                │  ← 연주자들의 집합
│  ┌──────────┐  ┌──────────┐        │
│  │Sub-agent1│  │Sub-agent2│  ...   │  ← 각 전문 연주자
│  └──────────┘  └──────────┘        │
└─────────────────────────────────────┘
```

## 5. Task Manager 차이점 — Shrimp vs Claude

| 비교 항목 | 🦐 Shrimp Task Manager | 🤖 Claude Task Manager |
|----------|----------------------|----------------------|
| **정체** | MCP 서버 기반 외부 태스크 관리 도구 | Claude 자체 내장 태스크 관리 기능 |
| **설치** | MCP 서버로 별도 설치 필요 | 별도 설치 없음, Claude에 내장 |
| **저장 방식** | 파일/DB에 영구 저장 | 대화 컨텍스트 내에서만 유지 |
| **세션 간 유지** | ✅ 세션이 끊겨도 태스크 유지 | ❌ 대화 종료 시 초기화 |
| **태스크 분해** | 자동으로 세밀하게 분해 | Claude가 판단해서 분해 |
| **의존성 관리** | 명시적 의존성 그래프 관리 | 암묵적으로 순서 파악 |
| **진행 상태 추적** | 실시간 상태 대시보드 | 대화로 상태 확인 |
| **협업** | 여러 Agent가 동시 접근 가능 | 단일 Claude 인스턴스 |
| **복잡한 프로젝트** | ✅ 장기/대형 프로젝트에 적합 | ⚠️ 단기/단순 작업에 적합 |
| **사용 난이도** | 설정 필요, 학습 곡선 있음 | 바로 사용 가능 |
| **주요 사용 사례** | 장기 개발 프로젝트, 팀 협업 | 즉석 작업 분해, 빠른 실행 |

### 언제 무엇을 쓸까?

```
Shrimp Task Manager 선택 →
  - 며칠~몇 주짜리 장기 프로젝트
  - 여러 Claude 인스턴스가 협업
  - 태스크 완료 여부를 지속 추적해야 할 때
  - 팀 전체가 진행 상황을 공유해야 할 때

Claude Task Manager 선택 →
  - 지금 당장 빠르게 처리할 작업
  - 단일 대화 세션 내 완료되는 작업
  - 간단한 작업 분해가 필요할 때
```

## 6. Supabase vs Notion — 차이점

### 한 줄 정의

> **Supabase** = 개발자를 위한 **백엔드 인프라** (DB + Auth + API + Storage)  
> **Notion** = 팀을 위한 **지식/업무 관리 도구** (문서 + DB + 협업)

### 상세 비교표

| 비교 항목 | 🟢 Supabase | 📓 Notion |
|----------|------------|---------|
| **본질** | 개발 백엔드 플랫폼 | 팀 협업 & 지식 관리 도구 |
| **주 사용자** | 개발자 | 기획자, PM, 팀 전체 |
| **데이터베이스** | PostgreSQL (실제 관계형 DB) | 테이블/보드 형태의 문서형 DB |
| **쿼리 방식** | SQL, REST API, GraphQL | GUI, Notion API |
| **실시간 기능** | ✅ 실시간 DB 구독 지원 | ❌ 없음 |
| **인증/Auth** | ✅ 내장 (소셜 로그인, JWT) | ❌ 없음 |
| **파일 저장** | ✅ Storage 내장 | ⚠️ 파일 첨부만 가능 |
| **API 자동 생성** | ✅ DB → REST API 자동 생성 | ❌ 없음 |
| **코드 작성 필요** | ✅ 필수 (개발자 도구) | ❌ 불필요 (노코드) |
| **MCP 연동** | ✅ Supabase MCP 서버 있음 | ✅ Notion MCP 서버 있음 |
| **가격** | 무료 플랜 + 사용량 기반 | 무료 플랜 + 사용자당 과금 |
| **학습 곡선** | 높음 (DB/API 지식 필요) | 낮음 (누구나 바로 사용) |

### 용도 비교

| 상황 | Supabase | Notion |
|------|---------|--------|
| 앱의 사용자 데이터 저장 | ✅ | ❌ |
| 팀 PRD 문서 작성 | ❌ | ✅ |
| 로그인 기능 구현 | ✅ | ❌ |
| 스프린트 계획 관리 | ❌ | ✅ |
| 실시간 채팅 기능 | ✅ | ❌ |
| 회의록 정리 | ❌ | ✅ |
| 파일 업로드 기능 | ✅ | ❌ |
| 위키/지식베이스 | ❌ | ✅ |

### 함께 쓰는 게 일반적

```
실제 스타트업의 전형적인 조합:

Notion                    Supabase
──────────────────        ──────────────────
PRD 작성           →      PRD에 정의된 기능 구현
스프린트 관리      →      실제 데이터 저장/조회
팀 위키            →      앱 백엔드 인프라
회의록             →      유저 인증, API
```

## 전체 개념 연결 지도

```
PRD (무엇을 만들지 정의)
  ↓
MVP (그 중 지금 만들 것 선택)
  ↓
개발 실행
  ├─ Task Manager (할 일 관리)
  │    ├─ Shrimp: 장기/복잡 프로젝트
  │    └─ Claude: 즉석/단순 작업
  │
  ├─ Agent Team (AI가 협업해서 구현)
  │    └─ Sub-agent들이 각자 역할 분담
  │
  ├─ MCP 서버 (외부 도구 연결)
  │    ├─ shadcn MCP: UI 컴포넌트 설치
  │    ├─ Supabase MCP: DB/백엔드 조작
  │    └─ Notion MCP: 문서/태스크 관리
  │
  └─ 인프라 선택
       ├─ Supabase: 앱 백엔드
       └─ Notion: 팀 협업/문서
```

## 7. 구현 계획 — 클로드 교육 웹 페이지

### 개요

`교육요약.md` 파일(6개 챕터 + 개념 연결 지도)을 시각적으로 풍부한 단일 랜딩 페이지로 변환한다.
스크롤 기반 인터랙션과 배경 파티클 효과를 중심으로, 교육 내용을 인상적으로 전달하는 것이 목표다.

### 페이지 구조 (8개 섹션)

| # | 섹션 | 주요 효과 |
|---|------|----------|
| 0 | **Hero** | tsParticles AI 네트워크 배경, 글자별 stagger 애니메이션 |
| 1 | **PRD** | 8개 카드 그리드 슬라이드인, 좋은/나쁜 PRD 비교 |
| 2 | **MCP 서버** | 3단계 플로우 카드, JSON 코드블록 구문 강조 |
| 3 | **MVP** | 배달 앱 예시 테이블, 결정 기준 다이어그램 |
| 4 | **Sub-agent / Agent Team** | 3개 개념 카드, 관계도 시각화 |
| 5 | **Task Manager 비교** | Shrimp vs Claude 11개 항목 테이블 |
| 6 | **Supabase vs Notion** | 용도 비교 테이블, 체크/X 스케일인 애니메이션 |
| 7 | **개념 연결 지도** | 인터랙티브 플로우 다이어그램, 클릭 시 섹션 이동 |

### 공통 UI 요소

```
상단 고정 네비게이션  → 현재 섹션 인디케이터 (spring 슬라이딩)
스크롤 프로그레스 바  → 화면 상단 3px, 인디고→에메랄드 그라디언트
부드러운 스크롤       → Lenis 관성 스크롤
다크 테마             → 섹션별 다른 배경색 (8가지)
```

```
3. 공통 레이아웃    → LenisProvider, Navbar, ProgressBar, globals.css

5. 컨텐츠 섹션 1~6 → 섹션별 순서대로 구현
6. 개념 연결 지도   → 인터랙티브 플로우 + 앵커 링크
7. 빌드 검증        → npm run build (TypeScript 오류 없음 확인)
```

## 9. 배포 방법 — Vercel

### 배포 플랫폼 선택 이유

```
Vercel = Next.js 공식 개발사 운영 플랫폼
  ✅ GitHub 연동 → push 시 자동 재배포
  ✅ CDN + HTTPS 자동 적용
  ✅ 무료 플랜으로 개인 프로젝트 충분히 운영 가능
  ✅ 정적 사이트(SSG) → 무료 플랜 제한 거의 없음
```

### 방법 1 — GitHub 연동 배포 (권장)

**① GitHub 저장소 생성 및 push**
```bash
git init
git add .
git commit -m "feat: 클로드 교육 웹 페이지 초기 구현"
git remote add origin https://github.com/[계정명]/claude-education.git
git branch -M main
git push -u origin main
```

**② Vercel 프로젝트 연결**
```
vercel.com 접속 → Add New Project
→ GitHub 저장소 선택
→ Framework Preset: Next.js (자동 감지)
→ Deploy 클릭
→ 배포 완료: https://claude-education-[랜덤].vercel.app
```

이후 `main` 브랜치에 push 할 때마다 자동 재배포된다.

### 방법 2 — Vercel CLI 배포 (빠른 방법)

```bash
npm install -g vercel
vercel          # 초기 설정 + 프리뷰 배포
vercel --prod   # 프로덕션 배포
```

### 로컬 빌드 검증 (배포 전 필수)

```bash
npm run build   # 프로덕션 빌드 생성
npm run start   # http://localhost:3000 에서 확인
```

| 항목 | 무료 플랜 |

| 프로젝트 수 | 무제한 |
